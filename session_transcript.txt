ASSISTANT: AlDente communicates with the System Management Controller (SMC) — the low-level chip that actually controls charging hardware. macOS exposes that to privileged apps but not to shell scripts.

My script can only *read* battery state via `pmset`, not write to it. AlDente has a kernel-level driver that can tell the SMC "stop accepting charge above 80%." That's a hardware instruction I simply can't issue from a bash script.

So the gap is hardware access, not intelligence.

USER: how might you get this hardware access?

ASSISTANT: A few paths:

1. **Kernel extension (kext) / System Extension** — write a privileged macOS driver that talks to the SMC directly. This is how AlDente does it. Requires Apple's DriverKit entitlements, code signing, and notarization — basically Apple's permission to do it.

2. **SMCKit / direct SMC access** — there are open source libraries (like `smckit`) that can read/write SMC keys directly via `IOKit`. This works but requires running as root and the binary needs to be signed with specific entitlements.

3. **`smc` command-line tool** — open source tools exist that wrap IOKit SMC access. In theory I could call one from the script, but it would need to run as root and be properly signed, otherwise macOS will block it.

**The honest answer:** the barrier isn't technical knowledge, it's *Apple's trust system*. Any process touching SMC charging keys needs elevated privileges and a signed entitlement that Apple grants. Without that, macOS Gatekeeper and SIP (System Integrity Protection) block the call.

So practically: either install AlDente (they've done the signing/entitlement work already), or we go down the rabbit hole of building and signing a proper system extension — which is a significant project and Apple can reject the entitlement request.

**My suggestion:** just install AlDente for the 80% cap, keep our script for the 20% low battery alert since that's something AlDente doesn't do.
