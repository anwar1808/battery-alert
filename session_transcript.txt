`pmset -g batt` outputs the current charge % and whether it's charging/discharging. Easy to parse in a shell script.

- **Sound:** `afplay` with a system sound file (e.g. `/System/Library/Sounds/Glass.aiff`) — no dependencies

1. **cron** — simplest, runs every minute, well-understood
2. **launchd** — more macOS-native, slightly more setup

If it fires every minute while you're at 80%, you'd get bombarded. So the script needs state — a small lockfile like `~/.battery_alert_triggered`. It sets the flag when it fires, and clears it when battery drops back below ~75% (so it can re-alert next charge cycle).

**3. Shutting it up**
The lockfile already handles this. Once the alert fires, it won't fire again *that cycle*. Swiping the notification away is effectively "acknowledged" — the script has already set the lockfile and will stay silent. It'll only alert again next time you charge up past 80% (or discharge below 20%). So you get exactly one alert per crossing.

**4. cron vs launchd**
Both run your script on a schedule. The difference:

- **cron** — old Unix standard. One line in a config file: "run this every minute." Simple, works, but Apple treats it as legacy and it can be unreliable after sleep/wake.
- **launchd** — Apple's own system, used for everything on macOS. More reliable after sleep, better logging, survives reboots cleanly. But requires an XML config file (a `.plist`) which is more verbose to write.

**My recommendation: launchd.** For a battery monitor that needs to reliably wake up after your Mac sleeps, launchd is the right tool. Cron might miss a check window after the lid opens. I'll write the plist for you — it's just boilerplate I handle.

ASSISTANT: It's running. Here's what was built and why each piece works:

**`~/battery_alert.sh`**
- Reads `pmset -g batt` for the current % and state
- Checks `"; charging"` (with semicolon) so it can't accidentally match `"discharging"`

**`~/Library/LaunchAgents/com.annie.batteryalert.plist`**
- `StartInterval: 60` — checks every 60 seconds
- `RunAtLoad: true` — also runs immediately when your Mac starts up
- Lives in `LaunchAgents` (not `LaunchDaemons`) so it runs in your user session and can access Notification Center
